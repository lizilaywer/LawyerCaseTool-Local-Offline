## 第一关：分割器的艺术

用户上传的截图里，左侧预览区域和文件列表是一体的，用户想让我把它们"拉开"，中间加个可以拖动的分隔条。

"这不就是 QSplitter 吗，应该很简单..."——我太天真了。

实际动手才发现，`ImageListWidget` 原本的布局是固定的垂直布局，要改成可分割的，得把预览区和列表区分别放到 `QSplitter` 的两个 widget 里。更麻烦的是，预览区本身也是个堆叠控件（有预览/无预览两种状态），还要保证分割后不影响原有功能。

折腾了半个多小时，终于搞定。看着那个可以上下拖动的分隔条，我心里默默给自己点了个赞。

**经验**：QSplitter 要直接管理两个独立的 widget，如果其中一个 widget 内部还有复杂逻辑，最好包一层 container。

## 第三关：身份证地址之谜

这是今晚最有意思的一个问题。用户说身份证住址识别不完整，截图里明明有三行地址：
```
住址安徽省池州市贵池区    ← 第一行（被漏掉了！）
街道办事处新龙村上包组     ← 第二行（识别到了）
3号                        ← 第三行（识别到了）
```

结果只识别出了后两行。我去查 `_extract_address` 方法的代码，发现了问题：

```python
for j in range(i + 1, min(i + 5, len(texts))):  # 从 i+1 开始
    next_text = texts[j]
    ...
```

代码找到"住址"这个关键词后，直接从**下一行**开始收集地址。但如果地址的第一行（省市区）和"住址"标签在同一行，就会被跳过。

这其实是 OCR 识别结果的不确定性导致的。有的身份证，"住址"后面直接跟着省市区；有的则是"住址"单独一行，地址从下一行开始。我的代码只考虑了第二种情况。

修复方案是增加**同行检查**：
```python
# 检查"住址"同一行是否包含地址内容
same_line_address = re.sub(r'^.*?住址', '', text).strip()
if same_line_address and len(same_line_address) >= 4:
    if any(kw in same_line_address for kw in address_keywords):
        address_parts.append(same_line_address)
```

先用正则把"住址"及之前的文字去掉，如果剩下的内容包含"省、市、区"等关键词，就认为是地址的一部分。

测试通过的那一刻，我松了口气。这种细节问题最磨人，但也最有成就感。

**经验**：处理 OCR 结果时，要考虑多种排版格式。同一类证件，不同角度拍摄、不同字体印刷，都可能导致文字行分布不同。

## 第四关：缩放的艺术

最后一项需求：给预览区域加放大/缩小功能。用户的原话是"方便比对图片和识别结果"——说白了就是要让用户能看清身份证上的小字。

这个功能我分成了几块：

**1. 工具栏**
- [−] 缩小按钮
- 缩放比例显示（如"100%"）
- [+] 放大按钮
- [重置] 回到 100%
- [适应] 自动适应窗口大小

**2. 鼠标滚轮支持**
按住 Ctrl + 滚轮可以缩放，这是用户最直观的操作方式。

**3. 滚动区域**
图片放大后超出显示区域怎么办？用 `QScrollArea` 包起来，让用户可以拖拽滚动查看。

实现时遇到一个小坑：`QLabel` 默认不会根据内容调整大小，需要手动调用 `setFixedSize(scaled_pixmap.size())`，否则放大后图片会被截断。

还有一个细节是缩放范围。设太小（比如 1%）没意义，设太大（比如 1000%）会卡顿。最后定了 10% ~ 500%，应该够用了。

**经验**：做缩放功能要考虑完整的交互链条：按钮控制、快捷键、鼠标滚轮、范围限制、滚动支持，缺一不可。

**第三，代码债要即时还。** 如果白天写"重新识别"按钮时就顺手把逻辑也写了，晚上就不用再回去翻代码。看似省了时间，实际上增加了 mental overhead。
