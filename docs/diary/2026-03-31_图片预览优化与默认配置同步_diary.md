## 一、图片预览缩放控件

### 起因

前两天用电子化归档功能的时候，打开一张图片，预览区下方空空如也——PDF 有放大缩小翻页一整套工具栏，图片什么都没有。缩放全靠鼠标滚轮？不，连滚轮缩放都没有。只能干瞪眼看原图。

用户反馈原话：在电子化归档界面中，如打开图片，在中间预览区域下方，没有与pdf一致的放大缩小、适应窗口等按钮及功能。

### 思路

打开 `archive_preview.py` 一看，逻辑很清晰：`_preview_pdf()` 显示工具栏，`_preview_image()` 直接 `_pdf_toolbar.setVisible(False)` 把工具栏藏了。工具栏本身是 PDF 和图片共用的，缩放逻辑（fit_mode、zoom level）也是共用的。

所以思路很简单：
1. 图片预览不再隐藏工具栏
2. 隐藏翻页相关控件（◀▶按钮、页码输入框），因为图片就一张
3. 复用 PDF 的缩放逻辑，新增 `_render_image()` 方法处理图片缩放渲染

### 实现

关键改动：
- 把翻页控件包进 `_page_nav_widget`，方便整体 show/hide
- 新增 `_render_image()` 方法，根据 fit_mode（width/fit/original/custom）计算缩放比例
- 修改 `_on_pdf_zoom_in/out/fit_mode()` 三个回调，根据 `_current_type` 分发到 `_render_image()` 或 `_render_pdf_page()`

这样图片预览就拥有了和 PDF 完全一致的缩放体验：放大、缩小、适应宽度、适应窗口、原始大小，还自动隐藏了无意义的翻页按钮。

### 经验

> 复用比重写好。PDF 工具栏的基础设施已经很完善了，图片只需要"借用"并做少量裁剪，没必要另起炉灶。关键是找到一个合适的抽象粒度——把翻页控件封装成一个 widget，show/hide 就行了。

## 二、对方当事人智能映射

### 起因

信息识别对话框里有个"是否为对方当事人"的复选框。勾选之后，识别出来的变量名会统一加 `opponent_` 前缀。比如识别到身份证上的姓名，正常匹配到 `client_name`，勾选后就变成 `opponent_client_name`，标签显示为"对方姓名(名称)"。

但用户指出了一个问题：如果当前模板是"民事案件简易模板(被告)"，对方就是原告。这时候 `opponent_client_name` 其实就应该直接填到"原告名称"那个变量里，而不是凭空多出一个 `opponent_client_name` 变量。

同理：
- 民事被告 → 对方姓名 → 原告名称 (`plaintiff_name`)
- 行政被告 → 对方姓名 → 原告名称 (`plaintiff_name`)
- 劳动仲裁被申请人 → 对方姓名 → 申请人姓名 (`applicant_name`)
- 商事仲裁被申请人 → 对方姓名 → 申请人名称 (`applicant_name`)

### 实现

在 `InfoExtractionDialog` 类中新增了一个类属性 `OPPONENT_DIRECT_MAP`：

```python
OPPONENT_DIRECT_MAP = {
    "civil_simple_002": "plaintiff_name",
    "admin_simple_002": "plaintiff_name",
    "labor_simple_002": "applicant_name",
    "commercial_simple_002": "applicant_name",
}
```

然后在 `_on_apply_to_template()` 和 `_save_and_apply()` 中，当勾选"对方当事人"且模板在映射表中时，`client_name` 的值直接赋给目标变量，而不是创建 `opponent_client_name`。

需要给 `InfoExtractionDialog.__init__` 加一个 `template_id` 参数，`main_window.py` 调用时传入当前选中的模板 ID。

### 经验

> 这个需求表面上是"改个变量名"，实际上涉及到对法律业务逻辑的理解。被告模板里"对方"就是原告，被申请人模板里"对方"就是申请人——这些隐含的业务规则如果不熟悉领域，很难自己推导出来。所以和用户确认业务需求永远比猜测靠谱。

> 另外一个小坑：构造函数加参数时要注意 `main_window.py` 的调用处也要同步修改，否则新参数永远收不到值，功能就会"静默失效"。

## 三、默认配置同步

### 起因

用户说"将现在的模板管理各项设置以及对应关联的word模板等设置，保存为默认值"。

这句话翻译成技术语言就是：把 `%APPDATA%/LawyerCaseTool/config/templates.json`（运行时配置）的内容写入 `src/config/default_templates.py`（代码默认值）。

为什么要这么做？因为 `default_templates.py` 里的默认模板是写代码时定义的"初始版本"，而用户在模板管理界面里做了大量调整：
- 修改了文件夹结构（加了"质量反馈意见卡"等文件）
- 关联了 Word 模板文件路径
- 通过 OCR 识别添加了新的变量（`opponent_*` 系列、`address`、`gender` 等）
- 调整了变量定义

这些调整都只保存在 `templates.json` 里，代码里的默认值还是旧的。新用户或重置后就会丢失这些精心配置的内容。

### 实现

写了一个 Python 脚本完成转换：
1. 读取 `templates.json`，修复 6 个模板的 category 错误（之前运行时配置里还是旧的 `criminal`）
2. 用递归函数把 JSON 转成 Python 字面量代码
3. 同时从 `config.json` 提取 `DEFAULT_APP_CONFIG`、`DEFAULT_GENERATION_CONFIG`、`DEFAULT_UI_CONFIG`、`DEFAULT_PINNED_CONFIG`
4. 写入 `default_templates.py`

踩了一个小坑：JSON 的 key `"global"` 是 Python 关键字，转成 Python 代码时必须加引号。第一版脚本用的 `k.isidentifier()` 判断，没覆盖到关键字的情况，生成的代码语法错误。改成始终用 `repr(k)` 就好了。

顺便把 `config_manager.py` 里的 `_get_default_config()` 方法也改了——之前它自己硬编码了一套默认值，和 `DEFAULT_*_CONFIG` 常量内容略有出入（比如 `default_output_dir` 一个是空字符串一个是 `~/案卷`）。现在直接引用常量，单一数据源，不会再出现不一致。

### 经验

> "默认值应该是什么"这个问题比看起来复杂。`default_output_dir` 保留用户本机路径 `C:\Users\49144\案卷`？不对，新用户的电脑上这个路径不存在。保留空字符串？`_get_default_config()` 又会用 `Path.home() / "案卷"` 覆盖。最终决定：代码默认值用空字符串（通用），`_get_default_config()` 引用同一套常量，首次运行时再根据用户机器动态设置。

> 另外一个值得注意的点：`templates.json` 里的 `template_path` 包含本机绝对路径（指向 Word 模板文件），如果项目要分发给其他用户，这些路径需要改为相对路径。目前暂时保留原样，用户后续可以调整。
