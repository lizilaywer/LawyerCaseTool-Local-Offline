## 发现的问题

### 严重问题 😱

审查过程中，我发现了一些让我"心跳加速"的问题：

**1. 重复的方法定义**
```python
# template_manager.py 中 _on_browse_template 方法定义了两次！
def _on_browse_template(self) -> None:
    ...
# ... 几十行后 ...
def _on_browse_template(self) -> None:  # 又来一次？
    ...
```
这就像是写了两个相同的方法名，Python 会静默地使用后一个定义覆盖前一个。这种问题在运行时不会报错，但会让代码变得混乱。

**2. 版本号不同步**
```python
# app.py
self._qt_app.setApplicationVersion("1.0.0")  # 这是啥？不是 1.1.3 吗？

# main_window.py 关于对话框
<p>版本: 1.0.0</p>  # 也不对
```
版本号不一致会让用户困惑，也会影响问题排查。

**3. 文件名中的中文引号**
```python
"name": '0"{{client_name}}"委托合同.docx'
```
中文引号 `"` 在 Windows 文件系统中是非法字符！虽然后面有 `sanitize_filename()` 函数处理，但数据源头就不应该有问题。

**4. 单例模式没有线程安全**
```python
def __new__(cls):
    if cls._instance is None:
        cls._instance = super().__new__(cls)
    return cls._instance
```
这个单例实现在多线程环境下会有竞态条件。虽然这个桌面应用目前不太可能用到多线程，但作为基础代码应该写得更健壮。

### 性能问题 🐢

**1. 模板扫描没有缓存**
每次调用 `get_available_templates()` 都会扫描整个文件系统。如果有 30+ 个模板文件，频繁调用会明显卡顿。

**2. 变量列表频繁重建**
搜索变量时，每次都删除所有控件再重新创建。这就像每次搜索都要把房子拆了重建一样浪费。

**3. 配置文件频繁写入**
每次 `set()` 都会写磁盘。如果连续设置 10 个配置项，就会写 10 次磁盘。

### 安全问题 🔒

**1. 路径遍历攻击风险**
```python
def resolve_template_path(self, relative_path: str):
    # 如果有人传入 "../../../../etc/passwd" 会怎样？
```
没有验证路径的安全性，理论上可以访问系统任意文件。

**2. 配置文件没有验证**
加载 JSON 配置时没有检查结构是否正确，格式错误的配置可能导致程序崩溃。

### 有趣的修复过程

**单例模式的双重检查锁**
```python
_lock: threading.Lock = threading.Lock()

def __new__(cls):
    if cls._instance is None:
        with cls._lock:
            if cls._instance is None:  # 再次检查
                cls._instance = super().__new__(cls)
    return cls._instance
```
这是一个经典的多线程安全模式。第一次检查是为了性能（避免每次都加锁），锁内的第二次检查是为了安全。

**模板扫描缓存**
```python
def get_available_templates(self, category: str = "", use_cache: bool = True):
    cache_key = category or "__all__"

    if use_cache:
        with self._cache_lock:
            if cache_key in self._templates_cache:
                cached_data, timestamp = self._templates_cache[cache_key]
                if time.time() - timestamp < self.CACHE_TTL:  # 30秒有效期
                    return cached_data.copy()
```
简单但有效的缓存策略：30 秒内重复调用直接返回缓存数据，同时提供 `clear_cache()` 方法在模板文件变更时手动刷新。

**批量更新的上下文管理器**
```python
with config_manager.batch_update():
    config_manager.set('key1', 'value1')
    config_manager.set('key2', 'value2')
    config_manager.set('key3', 'value3')
# 退出时自动保存，只写一次磁盘
```
Python 的上下文管理器真是优雅的设计模式！

## 经验总结

### 代码审查的收获

1. **文档很重要，但代码才是真相**
   文档说版本是 1.1.3，但代码里还是 1.0.0。永远要相信代码，而不是文档。

2. **防御性编程是必要的**
   不要假设输入一定是合法的。路径、文件名、配置数据都需要验证。

3. **性能问题往往藏在细节里**
   一次文件扫描不慢，但频繁扫描就会卡。一个控件创建不慢，但频繁创建销毁就会卡。

4. **单例模式要考虑线程安全**
   即使当前不需要多线程，也要为未来留有余地。

5. **异常处理要具体**
   `except Exception` 看起来省事，但会掩盖真正的问题。区分 `FileNotFoundError`、`PermissionError` 等具体异常，才能给出有意义的错误提示。

### 开发流程的思考

这次审查让我意识到，代码审查不应该只在项目后期做。定期的小规模审查比一次性大规模审查更有效。建议：

- 每个 PR 合并前做简单审查
- 每周做一次模块级别的审查
- 每月做一次全面审查
