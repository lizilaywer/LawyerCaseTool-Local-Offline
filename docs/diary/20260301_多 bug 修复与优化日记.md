今天是三月的第一天，春意渐浓，我的心情也随着代码的修复而变得舒畅起来。这个律师案卷工具项目已经陪伴我一段时间了，看着它从最初的概念逐渐成长为一个实用的桌面应用，心里还是有点小成就感的。

### 早晨：一个 "✓" 引发的思考

打开终端，看到日志里那些 `委托代理合同.docx✓` 这样的文件名，我愣了一下——这不是预览界面里用来标记"模板关联成功"的符号吗？怎么跑到导出的文件名里去了？

仔细一想，这其实是个很典型的"显示数据"和"业务数据"混淆的问题。预览界面上为了用户体验，加个 "✓" 让用户一眼就知道哪些文件已经关联了模板，这是个好设计。但问题是，保存配置的时候，我直接把显示文本存进去了...

```python
# 错误的做法
user_data["name"] = item.text(0)  # "委托代理合同.docx ✓"

# 正确的做法
user_data["name"] = item.text(0).replace(" ✓", "").replace(" 📎", "")
```

修复起来不难，但这个 bug 让我意识到：**任何为 UI 显示而添加的修饰，都应该在数据层剥离干净**。这算是一个经典的"关注点分离"教训。

### 中午：选择文字的"量子纠缠"

用户反馈说，在 Word 模板制作器里选择多个字，替换框只显示第一个字。

一开始我以为是编码问题——是不是 UTF-8 处理不对？还是 Unicode 分隔符的问题？

追踪代码后发现，问题出在 Qt 信号机制上：

```python
# 原来：copyAvailable 信号只在选择"状态"改变时触发一次
self._text_edit.copyAvailable.connect(self._on_selection_changed)

# 修改后：selectionChanged 信号在选区"内容"改变时都会触发
self._text_edit.selectionChanged.connect(self._on_selection_changed)
```

`copyAvailable` 只告诉你"有没有选中"，而 `selectionChanged` 才会告诉你"选中了什么"。这就像是：
- 前者只是通知你"有人敲门了"
- 后者才会告诉你"谁在敲门"

### 下午：变量不同步的烦恼

模板管理器和 Word 模板制作器是两个独立的界面，但它们应该共享同一份变量列表。

用户在模板管理器里添加了自定义变量，然后打开 Word 模板制作器——变量列表里没有！

追根溯源，发现 Word 模板制作器只从硬编码的 `DEFAULT_TEMPLATES` 加载变量
完全没有读取用户的自定义配置。

修复方案：
```python
def _load_default_variables(self):
    # 优先从用户配置读取
    config_manager = get_config_manager()
    templates = config_manager.get_templates()
    # ...合并变量
```

**经验：用户配置永远优先于默认配置。** 哪怕是"默认"的东西，也要给用户覆盖的机会。

### 傍晚：高亮格式的"传染"

用户说变量后面的普通文字也被高亮了。

这是一个 QTextCursor 的"特性"——当你用某种格式插入文本后
后续插入的文本会"继承"这个格式，除非你显式地设置新格式。

```python
# 插入变量时设置了高亮格式
cursor.insertText("{{variable}}", highlight_format)

# 插入普通文本时——居然还带着高亮！
cursor.insertText("普通文本")  # BUG: 继承了上面的格式

# 正确做法：显式重置格式
normal_format = QTextCharFormat()
cursor.insertText("普通文本", normal_format)  # OK: 使用默认格式
```

这让我想起了一个道理：**在 GUI 编程中，永远不要假设"默认值"会如你所愿。** 显式设置才是王道。

### 夜晚：撤销功能的崩溃

点击撤销按钮，程序崩溃了：`IndexError: pop from empty list`

看代码：
```python
def undo(self):
    if self._current_path:
        self.load_document(self._current_path)  # 这会清空 _history
        self._history.pop()  # BUG: history 已经空了！
```

`load_document()` 内部会调用 `_history.clear()`
然后再 `pop()`——这不是刻舟求剑吗？

修复很简单，删掉那行 `pop()` 就好。但这个小错误让我反思：**修改状态的方法不应该有隐藏的副作用**。如果 `load_document` 会清空历史
那方法名最好叫 `reload_document_clearing_history()` ——当然这是开玩笑的
但认真的做法是：在方法文档里明确说明副作用，或者干脆不产生副作用。

## 今日总结

修复了 5 个 bug：

1. 文件名包含显示标记 "✓"
2. 文本选择只获取首字

- **显示层和数据层的分离**
- **理解 API 的真实行为而非想当然**
- **用户配置优先于默认值**
- **显式设置胜于隐式继承**
- **注意方法的副作用**

写代码的时候，多问自己一句："这个操作真的会如我所想吗？"
