## 今日工作总览

今天是 v1.5.0 的主要开发日，从早上开始一直在搞一个比较大的新功能模块：案件管理 + 期限日历系统。这个功能在 PRODUCT_ROADMAP 里规划了很久，今天终于落地了。

核心思路很简单：律师用工具生成了案卷文件夹之后，需要一个地方统一管理所有案件，能看到哪些案件进行中、哪些待立案、哪些已结案，能看到每个案件的关键信息（案由、法院、案号），能看到期限提醒，能快速打开案件文件夹和文件。说白了，就是给案卷文件夹加一个"管理后台"。

## 一、数据层：CaseManager 单例

### 起点

第一个要解决的问题是：案件数据存在哪？

之前生成的案卷文件夹散落在用户电脑各处（C:\Users\xxx\案卷\张三劳动仲裁\ ...），没有统一的索引。每次打开软件，根本不知道用户之前生成过哪些案件。

我设计了两个存储方案：
1. **案件索引**：`%APPDATA%/LawyerCaseTool/config/cases.json`，存储所有案件的元数据（名称、路径、分类、状态、标签、期限等）
2. **案件速记**：`{案件文件夹}/.case/notes.md`，存储每个案件的临时笔记，跟着文件夹走（拷贝文件夹笔记也跟着走）

这个设计花了一点时间想清楚。速记文件为什么放在案件文件夹里面而不是 cases.json 里？因为律师经常会在电脑间拷贝案卷，或者用 U 盘带过去。如果速记存在 AppData 里，换个电脑就没了。放在文件夹里，带着走就行了。

### 实现

`case_manager.py` 是一个标准的单例模式（双重检查锁定，和 ConfigManager 保持一致），提供 CRUD、搜索、标签管理、期限管理这些方法。数据用 JSON 文件存储，读写用 `safe_write_json()` 原子写入（上次 v1.4.0 加固过的）。

### 一个小坑

`path_manager.py` 需要新增一个 `cases_file` 属性来返回 cases.json 的路径。这个改动本身很简单，但要确保路径目录存在——`config_dir` 目录在 `PathManager.__init__` 里已经创建了，所以 `cases_file` 指向 `config_dir / "cases.json"` 就行，不需要额外 mkdir。

## 二、UI 层：从卡片到面板到对话框

### 案件卡片 (CaseCard)

先做了最底层的组件——案件卡片。60px 高的紧凑型卡片，左侧 3px 分类色条（民事蓝、刑事红、行政橙...），然后是状态小标签 + 案件名称 + 标签 chips + 更新日期。

卡片的状态标签是后来加的。一开始做的版本没有状态显示，对照设计稿之后发现少了，才补上的。状态颜色用圆角边框小字标签：进行中=蓝色、待立案=黄色、已结案=灰色。

### 案件详情面板 (CaseDetailPanel)

这是今天改动最多的文件，前前后后改了三版。

第一版是最基础的实现：头部信息 + 文件树 + 底部期限列表。功能能用，但对照 PRODUCT_ROADMAP 设计稿发现差距很大。

第二版开始对齐设计稿：
- 从 `case["variables"]` 里提取关键字段（案由、法院、案号、委托人等），用 QGridLayout 做键值对网格展示
- 期限提醒从底部移到文件树上方，用大圆点 + 粗体天数 + 背景色区分紧急程度
- 文件树和预览并排，用 QSplitter 水平分割

第三版是用户截图评估后修复的 6 个 UI 问题（后面详细说）。

变量提取这块值得记一下。`case["variables"]` 是一个 dict，key 是模板变量名（比如 `Cause_of_Action`、`client_name`），value 是实际值。但这些变量名是给模板引擎用的，不是给人看的。所以需要一个映射表：

```python
VAR_DISPLAY = {
    "Cause_of_Action": "案由",
    "client_name": "委托人",
    "opponent_client_name": "对方当事人",
    "court_name": "法院",
    "case_number": "案号",
    ...
}
```

遍历 case["variables"]，如果 key 在 VAR_DISPLAY 里就显示成中文标签。不在的就跳过（模板变量太多了，不能全显示）。

### 案件管理对话框 (CaseManagerDialog)

左右分栏布局：左边案件列表，右边详情面板。

顶部一排搜索+筛选栏（搜索框 | 分类筛选 | 状态筛选 | 导入按钮），中间 QSplitter 左右分割，底部固定状态栏显示统计（共 N 个案件 | 进行中 X | 待立案 Y | 已结案 Z）。

案件列表按状态分组显示，每组前有一个带蓝色左边框的分组标题。这个分组逻辑是对照设计稿后加的，一开始是扁平列表。

这里踩了一个运行时 bug：`_load_cases()` 方法里用了 `c['text_muted']`，但 `c = COLORS` 只在 `_setup_ui()` 里定义了，`_load_cases()` 里没有重新定义。Python 不会跨方法共享局部变量，所以直接 `NameError`。用户把完整的 traceback 贴给我，一眼就定位到了。修复很简单，在方法开头加 `c = COLORS`。

### 期限日历 (CalendarDialog)

这个比较有趣。没有用 QCalendarWidget（样式太难定制），而是自己用 QPainter 画了一个日历。

画日历其实不难：
1. 计算当月第一天是星期几、当月有多少天
2. 用网格绘制日期格子（7列 x 6行）
3. 每个有期限的日期下面画几个小圆点（红=期限、蓝=开庭、黄=自定义）
4. 选中日期高亮，今天有特殊标记

右侧面板显示选中日期的期限详情列表。支持添加期限（标题+日期+类型）和添加开庭（标题+日期+时间+法院）。

## 三、生成自动注册

这个改动很小但很重要。

之前生成案卷后，案件文件夹就"放养"了，工具不记录它。用户想要管理这些案件，就得手动一个个导入。

改动在 `generation_dialog.py` 的 `_on_finished()` 里：生成成功后，自动调用 `CaseManager.register_case()` 把新案件注册到索引里，同时在案件文件夹下创建 `.case/notes.md` 速记文件。

这样每次生成案卷，案件管理里自动就有记录了。

## 四、文件树懒加载

案件详情面板里的文件树需要展示案件文件夹的内容。如果文件夹很深、文件很多，一次性加载会很慢。

参考了电子化归档的 `ArchiveFileTree`，给案件详情面板的文件树也加上了懒加载模式：
- 初始只加载根目录的直接子项
- 每个文件夹下面放一个占位子项
- 用户展开文件夹时，删除占位项，读取实际子目录并加载

这样即使一个案件有几百个文件，打开时也只加载第一层，体验流畅。

## 五、UI 评估与修复

下午的时候，用户截了一张运行中的界面截图发过来，让我"自己作为一名 UI 设计师评估一下"。

我仔细看了截图，对照 PRODUCT_ROADMAP 设计稿，找出了 6 个问题：

1. **筛选栏占了两行**：设计稿是单行，但搜索框+分类筛选+状态筛选+导入按钮挤在一起，自动换行了。合并成一行，中间用竖线分隔符隔开。

2. **分组标题不够醒目**：分组标题就是一行普通文字。加了蓝色左边框（`border-left: 3px solid accent`），和设计稿一致。

3. **缺少案件核心信息**：详情面板只显示了案件名称、分类、创建时间。缺少案由、法院、案号等关键字段。加了 VAR_DISPLAY 映射，从 variables 里提取并以键值对网格展示。

4. **"+ 添加"按钮位置不对**：期限区域的添加按钮和标题挤在一起。单独设置样式，和区域标题保持距离。

5. **预览区占位文字有问题**：预览区显示的占位文字格式乱了。清理了多余的样式和文字。

6. **速记区域不明显**：案件速记编辑器需要更明显的区域分隔。确保有标题和工具栏。

修完这些问题，界面确实好看多了。不过用户也说"目前还是有很多问题"，只是要求先暂停。

## 六、踩坑记录

### 坑 1：中文引号嵌套

今天踩了三次同一个坑。Python 代码里写了这样的字符串：

```python
label = QLabel("暂无案件\n可点击"导入文件夹"添加")
```

中文引号 `""` 和 Python 字符串的 `""` 混在一起，语法分析器分不清哪对是 Python 的、哪对是中文内容的。运行直接 SyntaxError。

修复方法：外层改用单引号：

```python
label = QLabel('暂无案件\n可点击"导入文件夹"添加')
```

三个文件（case_manager_dialog.py、case_detail_panel.py）都踩了这个坑。以后写含中文引号的字符串，统一用单引号包裹。

### 坑 2：局部变量跨方法访问

```python
def _setup_ui(self):
    c = COLORS
    # ... 用 c 设置样式 ...

def _load_cases(self):
    # 这里用了 c['text_muted']，但 c 是 _setup_ui 的局部变量！
```

这个 bug 在开发时很难发现（IDE 不会报错，只有运行到那行才 NameError）。用户把 traceback 贴过来才定位到。以后每个需要 COLORS 的方法都要自己 `c = COLORS`，不能偷懒。

### 坑 3：设计稿和实现的差距

一开始觉得自己实现的已经"差不多"了，对照设计稿才发现差了很多。期限提醒的位置、文件树+预览的并排布局、底部状态栏……这些都是看了设计稿才发现缺的。

教训：先看设计稿，再写代码。不要凭感觉写，写完再对——这样会返工。

## 今日感悟

今天的开发节奏大致是：数据层 → 底层组件 → 面板 → 对话框 → 对照设计稿 → 截图评估 → 修复问题。基本上是一个"螺旋上升"的过程——先让它能用，再让它好看，再让它和设计稿一致。

有个感受挺深的：**UI 开发最怕"差不多"**。"差不多能用了"和"和设计稿一致"之间，往往隔着一轮又一轮的微调。这些微调单个看都很小（改个间距、加个边框、调个颜色），但积少成多就很花时间。所以下次做 UI 模块，应该一开始就严格对照设计稿，把布局、间距、颜色都先定好，再开始写代码，而不是先写功能再调样式。

另外，中文引号这个坑踩了三次，说明需要建立代码规范：**所有含中文引号的字符串，外层统一用单引号**。这个要记到记忆文件里。

用户说"目前还是有很多问题，但是先暂停"。这说明 UI 还有打磨空间，但功能框架已经搭好了。下次继续的时候，重点是细节打磨——间距、对齐、交互反馈这些。
