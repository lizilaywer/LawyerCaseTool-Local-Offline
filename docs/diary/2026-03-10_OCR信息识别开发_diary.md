# 2026年3月10日 开发日记：OCR信息识别功能

> 一个产品经理+程序员的自白

---

## 清晨：接受挑战

早上9点多，我正在整理项目文档，用户突然发来需求——要在主界面添加一个"信息识别"按钮，支持识别身份证、户口簿、护照等证件信息，还要能自动填充到案卷模板里。

我心想：这需求不小啊。但看着用户那详细的描述和标注好的截图，我能感受到他的迫切——律师们每天要手动录入大量当事人信息，太容易出错了。

"行，开干！"

---

## 上午：架构设计

我先画了个草图：
- 左侧：图片列表 + 预览
- 右侧：识别结果展示 + 字段编辑
- 底部：进度条 + 操作按钮

技术选型上，用户明确说要**PaddleOCR**，理由是"本地识别、保护隐私"。我理解，律师的当事人信息确实敏感，不能上传到云端。

但问题很快就来了——PaddleOCR 的依赖太重了。我装了 paddlepaddle、paddleocr、paddlex... 一大堆包，还遇到了 Windows 长路径问题。花了将近一个小时才装好。

**教训1**: 下次应该先评估技术方案的部署成本。

---

## 中午：第一个坑

终于装好依赖，跑起来了，结果启动时卡死。原来是 PaddleOCR 初始化时会自动下载模型（100MB+），而下载源在国外，慢得要死。

我加了个环境变量 `PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK=True`，又做了延迟初始化，才让程序能正常启动。

用户点击"信息识别"按钮，添加了身份证图片，点击"开始识别"——**报错**：`Unknown argument: use_gpu`。

原来 PaddleOCR 3.x 版本的 API 变了，不支持 `use_gpu` 参数。我改了代码，又报 `Unknown argument: show_log`... 改完又报 `No module named 'paddle'`...

**教训2**: PaddleOCR 3.x 和 2.x 的 API 不兼容，文档又没写清楚。

---

## 下午：技术方案大转向

我实在受不了了，向用户提议：**换 RapidOCR**。理由是：
- 模型内置，无需下载
- 体积小（40MB vs 200MB+）
- 基于 ONNX Runtime，稳定不折腾

用户很开明："用最强大的 RapidOCR，把 PaddleOCR 残留全部清除！"

我如蒙大赦，立刻执行：
1. `pip uninstall paddlepaddle paddleocr paddlex`
2. `pip install rapidocr-onnxruntime`
3. 删除 `~/.paddlex` 缓存目录
4. 重写 OCR 引擎代码

整个过程不到20分钟。RapidOCR 果然香——初始化只要2秒，识别速度还更快。

**教训3**: 技术选型不能只看名气，稳定性更重要。

---

## 傍晚：细节打磨

识别功能跑通了，但用户测了一下，发现姓名识别成了"别男"。我一看，是解析逻辑有问题——把"性别"中的"别"字当成了姓名的一部分。

这种细节问题最磨人。我改了好几次正则表达式，又是加关键字过滤，又是改匹配顺序。终于让"章鸿鹏"正确识别出来了。

用户又提了两个小需求：
1. 左侧要有图片预览（方便对照）
2. 结果要显示性别和民族字段

这两个需求都很合理。我花了1个小时实现了预览功能——单张图片时自动显示在顶部，选中不同文件时自动切换。

---

## 晚上：收尾与反思

晚上10点，功能基本完工。我整理了文件：
- 更新了 VERSION 到 1.2.0
- 更新了 CHANGELOG
- 写了这篇日记

回想这一天的开发，最深刻的体会是：

**技术选型要务实**。PaddleOCR 确实强大，但对于桌面应用来说，部署成本太高了。RapidOCR 虽然名气没那么大，但解决了实际问题。

**用户体验在细节**。图片预览、字段置信度显示、低置信度标红... 这些小功能让工具从"能用"变成"好用"。

**用户是最好的产品经理**。用户的反馈非常精准——"姓名识别错了"、"加个预览吧"，每个建议都击中要害。
