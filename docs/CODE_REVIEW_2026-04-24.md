## 发现的问题

### 1. 高风险：相对目录未阻止路径穿越（已修复）

位置：`src/core/court_sms_service.py:93-97`

问题：

`_sanitize_relative_path()` 会保留 `..` 片段，随后 `base_path / Path(safe_relative)` 可把法院文书保存到案件目录之外。

风险：

如果相对目录中包含 `../`，程序可能把法院文书保存到用户未预期的位置，破坏“只能保存到指定案件目录内”的安全边界。

建议：

- 拒绝 `.`、`..`、绝对路径。
- 在 `mkdir` 前使用 `resolve()` 校验目标目录仍在 `base_dir` 内。
- 给该路径校验增加单元测试。

修复记录：

- `_sanitize_relative_path()` 已拒绝绝对路径、`.`、`..` 和无效片段。
- 保存目录创建前会校验最终路径仍位于用户选择的 `base_dir` 内。
- 已增加路径穿越回归测试。

### 2. 高风险：远端文件名未清洗（已修复）

位置：`src/core/court_sms_service.py:386-392`

问题：

法院接口返回的 `c_wsmc`/`c_wjgs` 直接组成 `document.name`，下载时拼到 `session_dir / document.name`。

风险：

如果远端返回的文件名包含路径分隔符、`..` 或非法字符，下载文件可能写出暂存目录，或在 Windows 上保存失败。

建议：

- 使用 `Path(name).name` 去掉目录部分。
- 统一使用 `sanitize_filename()` 清理文件名。
- 限制扩展名白名单，例如 `pdf/doc/docx/jpg/png/txt`。
- 保存前校验最终路径仍在 `session_dir` 内。

修复记录：

- 法院接口返回的文书名已统一清洗为普通文件名。
- 文书扩展名已限制在安全白名单内，未知扩展名回退为 `pdf`。
- 下载落盘前会再次清洗文件名并校验最终路径仍在暂存目录内。
- 已增加远端文件名清洗和下载目录边界回归测试。

### 3. 中风险：开庭事项会被逾期状态覆盖（已修复）

位置：`src/gui/calendar_dialog.py:522-526`

问题：

日期卡先判断是否逾期，再判断是否为开庭事项。当天早于当前时间的开庭会显示为“逾期”，而不是“开庭”。

风险：

用户可能在日历中看不到重要开庭事项的红色高亮，影响办案提醒准确性。

建议：

- 明确业务规则：开庭事项是否永远优先于逾期。
- 若开庭优先，则先判断 `type == hearing`，再判断逾期。
- 测试中注入固定时间，避免测试随着当前日期变化而失败。

修复记录：

- 日期卡状态判定已改为开庭事项优先，再判断逾期。
- 原失败的日历测试已通过。

### 4. 中风险：另存为失败会留下错误文件（已修复）

位置：`src/gui/archive_dialog.py:576-610`

问题：

另存为时先把原文件复制到最终输出路径，再做变量替换；如果替换失败，用户看到失败提示，但目标文件已经存在，且可能是未替换版本。

风险：

用户可能误以为导出的归档文档是正确的，实际里面仍是原始内容或半成品。

建议：

- 先写入临时文件。
- 所有变量替换成功后，再原子替换最终文件。
- 失败时删除临时文件，不污染用户选择的输出路径。

修复记录：

- 非覆盖原文件的另存为流程已改为先复制到临时文件。
- 变量替换和保存全部成功后才使用原子替换写入最终路径。
- 失败时会清理临时文件，避免留下半成品导出文件。

### 5. 中风险：大文件移动阻塞 UI（已修复）

位置：`src/gui/widgets/archive_file_tree.py:397-409`

问题：

文件树拖拽移动直接在 GUI 主线程中执行 `shutil.move()`。

风险：

移动大目录、网络盘文件或被系统占用的文件时，整个应用窗口可能无响应。

建议：

- 将移动操作放到 `QThread` 或任务队列。
- 显示进度、取消按钮和失败提示。
- 移动失败时保留原有结构并刷新文件树。

修复记录：

- 文件树拖拽移动已迁移到后台 `QThread` 执行。
- 移动完成后刷新结构、选中新位置并发送结构变更信号。
- 移动失败时在主线程提示错误，不再让主窗口长时间卡住。

### 6. 低风险：配置损坏时静默回退默认值（已修复）

位置：`src/config/config_manager.py:174-179`

问题：

配置或模板 JSON 解析失败后，程序直接使用默认配置，没有备份损坏文件，也没有明显提示用户。

风险：

用户可能感觉模板、设置突然“没了”，但其实原文件只是损坏或格式错误。

建议：

- 将坏文件重命名为 `.corrupt-时间戳.bak`。
- UI 弹出提示：已使用默认配置，并提供恢复路径。
- 增加配置恢复/导入工具。

修复记录：

- `config.json` 或 `templates.json` 损坏、格式错误、结构校验失败时，会先备份为 `.corrupt-时间戳.bak`。
- 备份后再创建新的默认配置或默认模板，避免静默覆盖用户原文件。
- 已增加配置和模板损坏备份回归测试。
