# 本地案件文件夹管理系统工作台重构方案

## 1. 产品定位

本项目后续的主线定位统一为“本地案件文件夹管理系统”。

它不是单一的模板生成器，也不是若干独立工具对话框的集合，而是一套以案件为中心、以本地文件夹为核心载体的律师桌面工作台。系统应围绕案件生命周期组织信息、文书、期限、费用、归档和辅助工具，让律师在一个主界面内完成日常工作切换。

本方案文档是后续 PySide6 实装的唯一设计基线。若旧文档、旧原型与本方案冲突，以本方案和新原型为准。

## 2. 目标与原则

### 2.1 重构目标

1. 统一产品形态：从“多个对话框工具”升级为“单窗口应用壳层 + 工作区标签页”。
2. 统一视觉语言：建立稳定、专业、适合法律场景的视觉系统。
3. 统一操作上下文：所有模块都能围绕当前案件进入，避免信息重复输入。
4. 统一信息结构：后续数据层按案件、期限、费用、文书、归档等实体收口。
5. 统一交付节奏：先完成文档与 HTML 原型，再据此推进桌面端重构。

### 2.2 设计原则

- 本地优先：所有核心能力以本地路径、本地模板、本地文件夹为基础。
- 案件优先：模板生成、OCR、归档和工具都服务于案件工作流。
- 工作台优先：页面切换应像同一款软件内的导航，而不是重新打开一个新窗口。
- 低认知负担：卡片、状态色、标题层级、交互控件保持统一。
- 可落地：原型中的页面结构和命名应直接映射到未来桌面应用。

## 3. 信息架构

### 3.1 主导航

主导航固定为以下 7 个一级入口：

1. 案件指挥台
2. 案件中心
3. 期限日历
4. 文书中心
5. 归档中心
6. 工具中心
7. 设置

### 3.2 工作区路由

HTML 原型与未来桌面工作区统一采用以下路由语义：

- `dashboard`
- `cases`
- `case:{id}`
- `calendar`
- `documents[:case_id]`
- `archive[:case_id|folder]`
- `tools`
- `settings`

### 3.3 页面关系

- 首页是固定的案件指挥台，不可关闭。
- 全局模块页可在工作区中单例复用。
- 案件页按 `case_id` 复用，不重复打开同一案件。
- 文书、归档、OCR 等能力既可以从全局入口进入，也可以从案件上下文进入，但必须复用同一页面与同一服务逻辑。

## 4. 统一应用壳层

### 4.1 壳层组成

所有主要页面都应复用同一套应用壳层：

- 左侧导航栏 `Nav Rail`
- 顶部全局栏 `Top Bar`
- 页面上下文栏 `Context Bar`
- 工作区标签栏 `Workspace Tabs`
- 主内容区 `Page Content`
- 底部状态栏 `Status Bar`

### 4.2 交互原则

- 一级页面切换只改变内容区，不更换整个窗口体验。
- 页面标题、副标题、当前案件上下文、快速操作入口在上下文栏统一呈现。
- 状态栏持续展示本地模式、模板目录、同步状态、最近保存信息等系统状态。
- 仅轻量编辑、确认和辅助输入保留弹窗；大模块全部收敛为可嵌入页面。

## 5. 页面说明

### 5.1 案件指挥台

案件指挥台是首页，也是产品的核心工作台。页面结构固定为：

- 顶部 KPI 区
  - 案件总数
  - 进行中案件
  - 已过期期限
  - 7 日内事项
  - 目录异常案件
  - 待收费用
- 中部可视化区
  - 案件状态分布
  - 案件类型分布
  - 近 30 天活动趋势
  - 期限时间线
- 右侧或下方工作队列
  - 今日事项
  - 本周到期
  - 最近更新
  - 目录异常
- 底部操作区
  - 新建案件
  - 导入案件文件夹
  - 新建期限
  - 打开文书中心
  - OCR 识别
  - 进入归档
  - 打开工具中心

### 5.2 案件中心

案件中心用于浏览、检索和筛选案件，应包含：

- 左侧搜索与过滤
- 中部案件列表、分组视图和摘要卡
- 右侧详情预览
  - 基本信息
  - 期限摘要
  - 费用摘要
  - 最近文件

### 5.3 单案件工作区

单案件工作区是案件详情与业务操作的承载页。内部子标签固定为：

- 总览
- 文件
- 期限
- 笔记
- 信息
- 费用

该页面必须体现案件上下文如何贯穿文书生成、OCR、归档与日常维护。

### 5.4 期限日历

期限日历以月视图为主，辅以周视图切换和任务侧栏。

状态表现必须统一：

- 过期未完成：灰色语义
- 即将到期：橙色语义
- 正常：蓝色或中性
- 已完成：文字删除线

### 5.5 文书中心

文书中心统一承载：

- 模板库
- 模板生成
- 最近文书

页面必须明确展示“全局模式”和“案件模式”的关系，强调只是上下文不同，不是两套系统。

### 5.6 归档中心

归档中心用于展示案件文件夹、归档状态、目录结构预览和批量处理区。既支持从全局进入，也支持带案件上下文进入。

### 5.7 工具中心

工具中心统一承载 OCR、截图合并、文档辅助、目录校验等工具，以统一卡片式布局呈现，避免工具像独立软件。

### 5.8 设置

设置页面重点呈现本地路径、模板目录、默认行为、界面偏好和未来的数据迁移说明。

## 6. 视觉系统

### 6.1 总体风格

视觉方向固定为“专业律所仪表盘”：

- 主色：深蓝 + 中性色
- 背景：浅纸感、轻纹理、低噪声
- 卡片：边界清晰、圆角适中、层级稳定
- 动效：少量、明确、有目的

第一版只做亮色主题，不做暗色主题。

### 6.2 组件基线

原型与后续实现统一采用以下组件概念：

- App Shell
- Nav Rail
- Page Header
- Context Bar
- Workspace Tabs
- Stat Card
- Chart Card
- Queue List
- Section Card
- Status Badge
- Deadline Chip
- Fee Summary Card
- Empty State

### 6.3 状态语义

- 正常：蓝色 / 中性
- 临近：橙色
- 过期：灰红
- 完成：文字删除线
- 异常：红色

## 7. 数据模型映射

HTML 原型只承担视觉与交互规格，不承担真实数据落库职责。但它的命名和信息结构必须贴近后续 SQLite 方向，避免原型和实现阶段重新翻译一遍。

演示数据字段至少包含：

- `cases`
- `deadlines`
- `fees`
- `activities`
- `documents`
- `archiveStatus`

后续正式数据层建议以 SQLite 为主存储，JSON 作为导入来源和兼容备份。

## 8. 交互约束

- 原型页面必须表现为同一款软件的不同页面，而不是多个独立网页。
- 所有页面复用同一套设计 token、壳层结构和导航系统。
- 不使用外部字体、图标 CDN 或 JS 图表库，保证本地静态打开即可使用。
- 原型交互仅限轻交互：
  - 页面导航高亮
  - 标签切换
  - 模拟搜索筛选
  - KPI 联动高亮
  - 队列展开/收起
  - 简单图表渲染

## 9. 实施阶段

### 阶段一：设计基线

- 完成本方案文档
- 完成多页面 HTML 原型
- 统一命名、页面边界和路由语义

### 阶段二：应用壳层重构

- PySide6 主窗口重构为统一壳层
- 各大模块改造成可嵌入页面
- 建立统一导航和标签页机制

### 阶段三：数据层收口

- 建立 SQLite Repository / Service 层
- 完成 JSON 到 SQLite 的迁移通道
- 工作台、案件页、期限、费用数据统一查询

### 阶段四：体验细化

- 补齐工作流细节
- 优化状态反馈和性能
- 增补会话恢复、视图状态保存等能力

## 10. 验收口径

### 文档验收

- `docs/WORKBENCH_REDESIGN_PLAN.md` 能独立说明产品定位、页面结构、数据方向和实施顺序。
- `PROJECT_RESUME.md` 能清晰指向本方案文档。

### 原型验收

- 所有原型页面在本地浏览器可直接打开。
- 原型共享同一设计系统和应用壳层，没有“拼起来”的观感。
- 首页完整呈现 KPI、图表、队列和快捷操作。
- 期限日历清楚表现过期灰色和完成删除线。
- 单案件工作区清楚表现案件上下文如何贯穿文书、期限、费用和归档。
- 文书中心和归档中心清楚表达“全局入口”和“案件入口”只是上下文不同。

### 实施约束

- 本轮只交付文档与 HTML 原型，不同步实现 PySide6 页面。
- 若后续实现需要新增页面或调整路由，应先回写本方案，再进入代码阶段。
