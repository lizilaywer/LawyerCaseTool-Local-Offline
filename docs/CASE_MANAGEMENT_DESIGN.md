# 案件文件夹管理系统本地化案件管理与期限管理 — 深度设计方案

> 核心理念：**文件系统即数据库，文件夹即案件，零导入零同步。**

> 本文档中的 `LawyerCaseTool` 路径命名保留为兼容标识，对外产品名称已统一为“案件文件夹管理系统”。

**编制日期**: 2026-04-02
**版本**: v1.0 设计稿

---

## 一、设计哲学：为什么"本地化"是正确方向

### 1.1 Web 端案件管理的真实痛点

律师每天的工作对象是**文件**——Word 文书、PDF 判决书、扫描件证据、照片。Web 端工具的致命缺陷：

| 痛点 | 场景 |
|------|------|
| **文件脱节** | 案件信息在网页上，但文件在本地 D:\案卷\ 里，两边来回切换 |
| **上传等待** | 每次要查看文件都得上传/下载，大 PDF 传半天 |
| **网络依赖** | 法院里没 WiFi、高铁上信号差、当事人办公室网络受限 |
| **加载慢** | 打开网页 → 登录 → 找案件 → 找文件 → 下载 → 打开，6 步才能看到文件 |
| **存储限制** | 云存储有容量上限，扫描件和视频证据体积大 |
| **隐私风险** | 案件材料上传到第三方服务器，律师保密义务怎么保障？ |
| **格式受限** | Web 预览 Word/PDF 效果差，排版错乱，无法编辑 |

### 1.2 本地化的独特优势

本工具的天然优势是**它就在文件旁边**。律师的案卷文件夹本来就在本地硬盘上，工具不需要"管理"文件——它只需要**知道文件在哪里，然后快速帮你打开它**。

```
Web 端流程（6步）:
打开浏览器 → 登录 → 找案件 → 找文件 → 下载 → 打开

本地化流程（1步）:
点击 → 打开
```

### 1.3 核心设计原则

| 原则 | 含义 |
|------|------|
| **文件系统即数据库** | 案件文件夹就是案件本身，不另建数据孤岛 |
| **零导入** | 文件不需要"导入"到工具中，工具直接读取磁盘上的文件 |
| **零同步** | 没有云端同步问题，修改即时生效 |
| **文件随身走** | 案件文件夹复制到 U 盘就是备份，发给同事就是移交 |
| **可离线** | 不依赖网络，任何环境都能工作 |
| **快速访问** | 单击预览、双击打开、右键操作，减少一切中间步骤 |

## 二、案件管理 — 详细设计

### 2.1 整体思路：不是"管理系统"，而是"超级书签"

不要把案件管理做成传统的"增删改查"系统。律师不需要另一个数据库来存案件信息——他们已经在用 Windows 资源管理器管理案卷文件夹了。

本工具要做的是：

1. **记住这些文件夹在哪** — 案件索引
2. **帮你快速找到它们** — 搜索和筛选
3. **帮你快速打开文件** — 一键打开、预览
4. **补充文件夹无法承载的信息** — 期限、备注、状态

**类比**：就像浏览器的书签功能。浏览器不"管理"网页，它只是帮你记住网址、快速打开。本工具不"管理"案件文件夹，它只是帮你记住路径、快速访问。

### 2.2 数据模型：案件索引

#### 存储位置

```
%APPDATA%/LawyerCaseTool/
├── config.json          # 已有：应用配置
├── templates.json       # 已有：模板配置
└── cases.json           # 新增：案件索引（轻量级 JSON）
```

**为什么用 JSON 而不是 SQLite**：
- 案件数量有限（律师同时处理的案件通常 20-50 个，一年累计 100-200 个）
- JSON 人类可读，可以直接编辑、可以 git diff、可以发给别人
- 与现有 config.json / templates.json 风格一致
- 不需要复杂查询（搜索用 Python 内存过滤足够快）

#### 案件索引结构

#### 关键设计决策

**字段分两层**：

- **自动提取层**（工具自动填充，零手动输入）：
  - `folder_path` — 案件文件夹的磁盘路径
  - `display_name` — 从文件夹名解析（`2026-03-31_张三_民事_借款纠纷` → `张三 诉 李四 — 借款纠纷`）
  - `client_name` — 从文件夹名或生成变量中提取
  - `case_type` — 从使用的模板 ID 推断（civil_simple_001 → civil）
  - `template_id` — 生成时使用的模板 ID
  - `created_at` — 文件夹创建时间
  - `last_opened_at` — 最近打开时间

- **手动补充层**（律师按需填写，非必填）：
  - `opponent_name` — 对方当事人
  - `cause_of_action` — 案由
  - `court` — 法院
  - `case_number` — 案号
  - `filing_date` — 立案日期
  - `notes` — 备注

**重要：所有手动字段都是可选的。** 律师可以只把工具当"书签"用，不填任何额外信息。也可以在收到案号后回来补充。

### 2.3 案件注册：自动 vs 手动

#### 方式一：生成时自动注册（零操作）

当前生成流程（`generation_dialog.py`）生成成功后，只做了两件事：
1. 打开文件夹
2. 弹出成功提示

**改进**：生成成功后，自动将新文件夹注册到案件索引中。

```python
# generation_dialog.py _on_finished() 中新增：
def _on_finished(self, result):
    if result.get("success"):
        root_path = result.get("root_path")
        # ... 现有逻辑 ...

        # 新增：自动注册案件
        from src.core.case_manager import get_case_manager
        case_manager = get_case_manager()
        case_manager.register_from_generation(
            folder_path=root_path,
            template_id=self._template_id,
            values=self._values  # 生成时填写的变量值
        )
```

注册时自动提取的信息：
- `folder_path` — 生成的根目录路径
- `display_name` — 从 `root_name` 解析
- `client_name` — 从变量 `{{client_name}}` 中取值
- `case_type` — 从 `template_id` 推断
- `template_id` — 直接传入
- `receive_date` — 从变量 `{{receive_date}}` 中取值
- `cause_of_action` — 从变量 `{{Cause_of_Action}}` 中取值
- `created_at` — 当前时间

律师**什么都不用填**，案件就自动出现在案件列表里了。

#### 方式二：手动添加已有文件夹

对于工具生成之前就已经存在的案卷文件夹：

```python
# 新增对话框 AddCaseDialog
def _on_add_existing_case(self):
    folder_path = QFileDialog.getExistingDirectory(
        self, "选择案卷文件夹"
    )
    if folder_path:
        case_manager.register_existing(folder_path)
```

注册时尝试从文件夹名中解析信息：
- `2026-03-31_张三_民事_借款纠纷` → 日期、当事人、案件类型、案由
- 如果文件夹名不符合命名规则，就用文件夹名作为 `display_name`，其他字段留空

#### 方式三：扫描目录批量注册

律师通常按年份或月份组织案卷文件夹：

```
D:/案卷/
├── 2026年/
│   ├── 2026-03-31_张三_民事_借款纠纷/
│   ├── 2026-04-01_王五_刑事_盗窃罪/
│   └── ...
├── 2025年/
│   └── ...
```

提供"扫描目录"功能，自动发现所有案卷文件夹并批量注册：

```python
def scan_directory(self, dir_path: Path):
    """扫描目录，自动发现并注册案卷文件夹"""
    for sub_dir in dir_path.iterdir():
        if sub_dir.is_dir() and not sub_dir.name.startswith('.'):
            # 尝试匹配文件夹名格式
            if self._is_case_folder(sub_dir.name):
                self.register_existing(sub_dir)
```

匹配规则（可扩展）：
- 以日期开头：`YYYY-MM-DD_...`
- 包含案件类型关键词：`民事`、`刑事`、`行政`、`仲裁`
- 不匹配的也允许注册，只是自动提取的信息较少

### 2.5 文件操作 — "超级文件管理器"

这是本地化的核心体验。案件详情的中间面板不只是"显示文件树"，而是提供**比 Windows 资源管理器更适合律师的文件操作**。

#### 文件操作矩阵

| 操作 | 触发方式 | 行为 | 技术实现 |
|------|---------|------|---------|
| **预览** | 单击文件 | 右侧面板显示预览（Word文本/PDF渲染/图片） | 复用 `archive_preview.py` |
| **打开** | 双击文件 | 用系统默认程序打开（Word/PDF阅读器/图片查看器） | `os.startfile(path)` |
| **定位** | 右键 → "在资源管理器中打开" | 打开 Windows 资源管理器并选中该文件 | `explorer /select,"path"` |
| **重命名** | 右键 → 重命名 | 文件树内联编辑，同步修改文件系统 | 复用 `archive_file_tree.py` 逻辑 |
| **新建文件** | 右键文件夹 → "新建文件" | 选择模板或创建空文件 | `QFileDialog` + `TemplateEngine` |
| **新建文件夹** | 右键文件夹 → "新建子文件夹" | 输入名称，创建目录 | `Path.mkdir()` |
| **删除** | 右键 → 删除 | 确认后删除文件/文件夹 | `Path.unlink()` / `shutil.rmtree()` |
| **复制路径** | 右键 → "复制路径" | 复制完整路径到剪贴板 | `QClipboard` |
| **拖拽移动** | 拖拽文件到另一个文件夹 | 在案件文件夹内移动文件 | `shutil.move()` |

#### "打开文件夹" — 最常用的操作

在案件详情顶部提供醒目的"打开文件夹"按钮：

```python
def _on_open_folder(self):
    """在 Windows 资源管理器中打开案件文件夹"""
    folder_path = self._current_case["folder_path"]
    os.startfile(folder_path)
```

同时提供"打开当前文件所在文件夹"：

```python
def _on_open_in_explorer(self):
    """在资源管理器中定位到当前选中的文件"""
    file_path = self._get_selected_file_path()
    subprocess.run(f'explorer /select,"{file_path}"')
```

#### 最近打开的文件

在案件列表底部或状态栏显示"最近打开的文件"，点击直接打开：

```
最近文件: 起诉状.docx (张三案) | 判决书.pdf (王五案) | 证据目录.docx (赵六案)
```

#### 案件间文件对比

律师经常需要对比不同案件的同类文书（如两份起诉状的格式差异）。提供"在新标签页中打开"功能，支持同时查看多个文件。

### 2.6 案件状态管理

#### 状态流转

```
收案 ──→ 立案 ──→ 审理 ──→ 判决 ──→ 结案归档
                                          ↓
                                    上诉/执行（可选）
```

简化为三个分组显示：

| 分组 | 对应状态 | 筛选条件 |
|------|---------|---------|
| 进行中 | `active` | 有 `filing_date` 或 `receive_date`，无 `close_date` |
| 待立案 | `pending` | 有 `receive_date`，无 `filing_date` |
| 已结案 | `closed` | 有 `close_date` |

**状态变更方式**：不是复杂的表单，而是在案件列表中右键 → "标记为已立案" / "标记为已结案"。

或者更简单：在案件信息面板中，状态是一个下拉选择框。

#### 结案归档流程

点击"结案归档"后：
1. 将状态改为 `closed`
2. 自动记录 `close_date`
3. 弹出归档对话框（复用现有 `ArchiveDialog`）
4. 归档完成后，案件从"进行中"移到"已结案"

### 2.8 搜索与筛选

#### 搜索范围

## 三、期限管理 — 详细设计

### 3.1 整体思路：轻量级，不过度设计

期限管理的核心需求很简单：**告诉我什么时间之前必须做什么事，提前提醒我。**

不需要复杂的甘特图、不需要团队协作、不需要移动端推送。律师需要的是：
1. 设一个日期 → 工具记住
2. 快到了 → 工具提醒
3. 过期了 → 工具标红

### 3.2 期限数据模型

期限作为案件的子数据，存储在 `cases.json` 中（见上文数据模型）：

```json
{
  "deadlines": [
    {
      "id": "d001",
      "title": "举证期限",
      "date": "2026-04-15",
      "type": "evidence",
      "related_date": "2026-03-31",
      "calc_rule": "filing_date + 15d",
      "status": "pending",
      "notes": "需要补充银行流水"
    }
  ]
}
```

#### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 唯一标识 |
| `title` | 是 | 期限名称（如"举证期限"、"上诉期限"） |
| `date` | 是 | 截止日期 |
| `type` | 是 | 类型：`evidence`(举证) / `appeal`(上诉) / `hearing`(开庭) / `payment`(付款) / `custom`(自定义) |
| `related_date` | 否 | 关联的案件日期字段（如 `filing_date`），用于自动计算 |
| `calc_rule` | 否 | 计算规则（如 `filing_date + 15d`），用于自动推算 |
| `status` | 是 | `pending`(待处理) / `completed`(已完成) / `overdue`(已过期) |
| `notes` | 否 | 备注 |

### 3.3 期限来源：三种方式

#### 方式一：手动添加

在案件详情面板的"期限提醒"区域，点击"添加期限"：

```python
def _on_add_deadline(self):
    """添加期限"""
    dialog = AddDeadlineDialog(self)
    if dialog.exec():
        deadline = dialog.get_data()
        self._current_case["deadlines"].append(deadline)
        self._save_case()
```

对话框字段：标题、截止日期、类型、备注。简单直接。

#### 方式二：自动计算（从关键日期推算）

当律师在案件信息中填写了"立案日期"后，工具自动计算关联期限：

```python
# 期限规则定义
DEADLINE_RULES = {
    "civil": {
        "filing_date": [
            {"title": "答辩期限", "offset_days": 15, "type": "defense"},
            {"title": "举证期限", "offset_days": 15, "type": "evidence"},
        ],
        "judgment_date": [
            {"title": "上诉期限", "offset_days": 15, "type": "appeal"},
            {"title": "生效日期", "offset_days": 15, "type": "effective"},
        ],
    },
    "criminal": {
        "filing_date": [
            {"title": "答辩期限", "offset_days": 15, "type": "defense"},
        ],
    },
    "administrative": {
        "filing_date": [
            {"title": "答辩期限", "offset_days": 15, "type": "defense"},
            {"title": "举证期限", "offset_days": 15, "type": "evidence"},
        ],
    },
}
```

**触发时机**：
- 律师在案件信息中填写/修改日期字段时
- 自动计算并提示"发现 2 个关联期限，是否添加？"
- 不自动添加（避免制造噪音），由律师确认

#### 方式三：从 OCR 识别结果提取

OCR 识别法院传票后，自动提取开庭日期并创建期限：

```
OCR 识别传票 → 提取开庭日期 → 自动创建开庭期限 → 提醒律师
```

## 五、关键技术实现细节

### 5.1 文件夹名解析器

从工具生成的文件夹名中自动提取案件信息：

```python
import re
from datetime import datetime

class FolderNameParser:
    """从案卷文件夹名中解析案件信息"""

    # 匹配格式: YYYY-MM-DD_当事人_案件类型_案由
    PATTERN = re.compile(
        r'^(\d{4}-\d{2}-\d{2})'     # 日期
        r'_([^_]+)'                   # 当事人姓名
        r'_(民事|刑事|行政|劳动仲裁|商事仲裁|非诉)'  # 案件类型
        r'_(.*)$'                     # 案由
    )

    TYPE_MAP = {
        "民事": "civil",
        "刑事": "criminal",
        "行政": "administrative",
        "劳动仲裁": "labor_arbitration",
        "商事仲裁": "commercial_arbitration",
        "非诉": "non_litigation",
    }

    @classmethod
    def parse(cls, folder_name: str) -> dict:
        match = cls.PATTERN.match(folder_name)
        if not match:
            return {"display_name": folder_name}

        date_str, client_name, type_str, cause = match.groups()
        return {
            "receive_date": date_str,
            "client_name": client_name,
            "case_type": cls.TYPE_MAP.get(type_str, "civil"),
            "cause_of_action": cause,
            "display_name": f"{client_name} — {cause}",
        }
```

### 5.3 期限检查器

```python
from datetime import datetime, timedelta

class DeadlineChecker:
    """检查即将到期和已过期的期限"""

    WARNING_DAYS = 7    # 红色预警天数
    NOTICE_DAYS = 30    # 黄色提醒天数

    @classmethod
    def check_all(cls, cases: list) -> list:
        """检查所有案件的期限，返回需要提醒的列表"""
        today = datetime.now().date()
        alerts = []

        for case in cases:
            for deadline in case.get("deadlines", []):
                if deadline.get("status") == "completed":
                    continue

                due_date = datetime.strptime(deadline["date"], "%Y-%m-%d").date()
                days_left = (due_date - today).days

                if days_left < 0:
                    urgency = "overdue"
                elif days_left <= cls.WARNING_DAYS:
                    urgency = "urgent"
                elif days_left <= cls.NOTICE_DAYS:
                    urgency = "notice"
                else:
                    continue  # 还早，不需要提醒

                alerts.append({
                    "case": case,
                    "deadline": deadline,
                    "days_left": days_left,
                    "urgency": urgency,
                })

        return sorted(alerts, key=lambda x: x["days_left"])
```

## 六、与 Web 端方案的对比

| 维度 | Web 端案件管理 | 本方案（本地化） |
|------|--------------|---------------|
| 文件打开速度 | 需上传/下载，3-10秒 | 即时，0.1秒 |

| 文件格式 | Web 预览，格式丢失 | 原生打开，格式完美 |

## 七、总结

### 核心设计决策

1. **不做"管理系统"，做"超级书签"** — 案件数据跟着文件夹走，工具只负责记住和快速访问
2. **零导入** — 文件不需要导入，直接读取磁盘
3. **生成即注册** — 用工具生成案卷后自动出现在案件列表，零额外操作
4. **文件操作优先** — 单击预览、双击打开、右键定位，比资源管理器更方便
5. **期限轻量化** — 不做复杂日历，做启动提醒 + 列表标记
6. **复用现有代码** — 文件树、文件预览、归档对话框全部复用，开发量可控

### 量化预期

| 指标 | 当前 | 改进后 |
|------|------|-------|
| 找到一个案件文件 | 切换到资源管理器 → 找文件夹 → 找文件（15秒） | 点击案件 → 点击文件（2秒） |
| 查看案件信息 | 翻文件夹、翻文件（30秒） | 案件详情面板即时显示（1秒） |
| 期限查看 | 靠记忆或外部日历 | 启动时自动弹窗提醒 |
| 新建案卷到可编辑 | 生成 → 打开文件夹 → 找文件 → 双击打开（10秒） | 生成 → 自动注册 → 点击文件打开（3秒） |
