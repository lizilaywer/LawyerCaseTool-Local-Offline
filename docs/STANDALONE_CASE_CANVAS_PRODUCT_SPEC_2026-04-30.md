# 独立软件项目文档：案件画布 / 材料导图工作台

## 1. 文档目的

本文档用于整理一个**独立于当前“案件文件夹管理系统”之外的新软件方向**。  
该软件暂不要求与现有项目集成，也不以复用当前 UI 结构为前提，而是从零定义一个更适合“案件材料全览、归类、预览、拆分、移动、标注”的本地桌面工具。

本文档目标是把需求从“想法”整理成**可落地、可分阶段实现、可评估成本**的项目方案。

---

## 2. 暂定产品名称

可选名称：

- `案件画布`
- `材料画布`
- `案件导图工作台`
- `证据画布`
- `案件全览台`

建议优先名称：

- **案件画布**

原因：

- 比“思维导图”更准确，不会让用户误以为只是笔记工具
- 比“节点编辑器”更贴近法律业务场景
- 可以容纳文件系统、PDF、笔记、标注、关系连线、拆分归类等多种操作

---

## 3. 产品定位

### 3.1 一句话定位

一个面向律师和案件材料整理场景的**本地化自由画布文件管理软件**，用“导图/节点画布”的方式直接操作真实文件系统，并支持大 PDF 的快速预览、拆分、归类和标注。

### 3.2 核心目标

不是做一个普通的思维导图软件，也不是做一个普通的文件管理器，而是做一个：

- 以**案件材料**为对象
- 以**自由画布**为操作空间
- 以**真实文件系统**为底层载体
- 以**PDF 预览、拆分、归类**为高频核心能力

的桌面软件。

### 3.3 用户真正要解决的问题

当前常规文件夹树在以下场景中体验很差：

- 案件目录层级太深，存在大量“套娃式”文件夹
- 材料量很大，尤其是扫描件、证据 PDF、庭审材料
- 需要一边看全局结构，一边移动、归类、拆分文件
- 需要快速知道“某份文件应该进哪个分支”
- 需要在视觉上快速理解案件材料结构，而不是反复点开文件夹

用户希望获得一种“上帝视角”：

- 一眼看到全局结构
- 在画布上直接操作文件与文件夹
- 随时预览、拆分、重组、标注
- 操作结果实时反映到真实磁盘目录

## 4. 产品形态定义

### 4.1 不是传统脑图

该产品不应做成传统的中心主题发散式脑图。  
真实文件系统是**单根、多层级、强父子关系**的数据结构，传统脑图不适合深目录和海量材料。

### 4.2 更接近的形态

更适合的形态是：

- `ComfyUI` 式自由画布
- `思维导图` 式全局可视化
- `文件管理器` 式真实操作能力
- `PDF 工作台` 式预览与拆分能力

可以理解为：

> “自由画布上的文件系统管理器 + 案件材料工作台”

### 4.3 推荐默认视图

推荐采用**左到右层级展开 + 自由拖动画布 + 可折叠节点**的混合结构：

- 文件夹节点：默认展开为主结构
- 文件节点：默认收纳于文件夹下，可按需展开
- 备注/标注节点：悬挂在文件或文件夹旁侧
- 关系连线：用于业务语义，不改变真实文件系统结构

---

## 5. 核心需求复述

以下为当前需求的准确收束版本。

### 5.1 结构可视化

- 导入某个案件目录后，将真实文件夹结构转换为画布上的节点结构
- 用户可以在画布上快速查看多层目录
- “套娃式”目录应能比传统树结构更容易理解
- 支持缩放、拖拽、折叠/展开、全局缩略导航

### 5.2 文件系统真实联动

- 画布上的文件夹节点对应真实磁盘文件夹
- 画布上的文件节点对应真实磁盘文件
- 在画布上新建文件夹分支，即在磁盘中创建对应文件夹
- 将文件拖动到另一个分支，即在磁盘中执行真实移动
- 重命名、删除、复制、粘贴等操作应最终反映到真实文件系统

### 5.3 打开与预览

- 双击文件节点：打开文件
- 双击文件夹节点：展开/折叠或在系统文件管理器中定位
- 单击文件节点后按空格：像 `Quick Look` 一样快速预览
- 预览应优先在软件内部完成，而不是强依赖系统能力

### 5.4 PDF 专项能力

针对超大 PDF，目标能力包括：

- 快速预览
- 按页/范围拆分
- 拆出多个 PDF 片段
- 将片段快速拖放到不同材料分支
- 边拆边归类，形成新的结构节点

### 5.5 标注与记录

除了文件系统节点，还需要支持：

- 备注节点
- 标签/颜色标注
- 对某份文件的处理说明
- 对某个分支的整理计划
- 对多个材料的逻辑关系连线

### 5.6 自由画布体验

用户期待的不是“只能看目录”，而是：

- 可以拖动画布
- 可以调整节点布局
- 可以局部整理结构
- 可以保留业务语义标注
- 可以从全局视角理解整个案件材料空间

---

## 6. 第一阶段不做什么

为了确保项目可落地，第一阶段必须克制。以下内容不建议在 MVP 一次性做完：

- 不做完整的多用户协作
- 不做云同步
- 不做完整版本控制系统
- 不做 OCR 全流程自动识别平台
- 不做法律知识图谱
- 不做复杂流程自动化引擎
- 不做完全无限自由的“任意节点都能成为文件父级”的结构

原因：

- 真实文件系统决定了文件只能有一个父目录
- 第一阶段应优先保证“稳定、可操作、可同步”
- 先把真实文件操作打通，比做酷炫自由画布更重要

---

## 7. 核心对象模型

建议区分两类结构：

### 7.1 真实结构层

真实文件系统对象：

- `WorkspaceProject`
- `FolderNode`
- `FileNode`
- `PdfDocument`

这一层是**真实源**，决定磁盘上的最终结果。

### 7.2 画布表现层

画布附加对象：

- `CanvasNodePosition`
- `CanvasGroup`
- `RelationEdge`
- `AnnotationNode`
- `SplitPreviewNode`
- `TagMarker`

这一层是**可视化与业务语义层**，不改变真实磁盘结构本体。

### 7.3 关键原则

必须明确：

> 文件系统是真实源，画布是操作与表达层。

不能让画布模型反过来完全主导文件系统，否则很快会出现结构混乱和同步灾难。

---

## 8. 推荐功能分层

### 8.1 MVP（第一阶段）

目标：先做“可视化 + 可打开 + 可预览 + 可移动”。

包含：

- 导入案件目录
- 目录转换为画布节点
- 缩放、平移、折叠/展开
- 双击打开文件/文件夹
- 空格快速预览
- 新建文件夹分支
- 文件拖放到目标分支并真实移动
- 节点基础备注

### 8.2 Phase 2（第二阶段）

目标：做“高效整理”。

包含：

- 批量多选
- 批量移动
- 批量复制/粘贴
- 节点颜色标识
- 文件夹模板分支快速创建
- 回收站/误操作恢复
- 外部文件系统改动检测同步

### 8.3 Phase 3（第三阶段）

目标：做“PDF 专项工作台”。

包含：

- 大 PDF 流畅预览
- 页缩略图
- 按页拆分
- 按区间拆分
- 拖拽页/片段生成新 PDF
- 拆分后直接归类到分支

### 8.4 Phase 4（第四阶段）

目标：做“业务语义画布”。

包含：

- 文件节点和备注节点自由连线
- 证据关系、案件脉络标注
- 时间线节点
- 任务节点
- 证据组/争点组可视化组织

## 9. 用户交互草案

### 9.1 主界面结构建议

推荐结构：

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部工具栏：项目 / 搜索 / 视图切换 / 预览 / 拆分 / 归类     │
├───────────────┬──────────────────────────────┬────────────┤
│ 左侧项目树/   │ 中间自由画布                  │ 右侧预览/   │
│ 导航索引      │ 文件夹节点、文件节点、备注    │ 属性/拆分区 │
│               │                               │            │
└───────────────┴──────────────────────────────┴────────────┘
```

### 9.2 推荐交互规则

- 单击：选中节点
- 双击：打开文件 / 展开文件夹
- 空格：预览当前文件
- 拖动节点：调整画布布局
- 拖动文件到文件夹节点：真实移动文件
- 右键：新建、重命名、复制、粘贴、删除、定位、标注
- 滚轮：缩放画布
- 按住空格拖动空白区：平移画布

### 9.3 预览建议

第一版建议做成软件内右侧预览面板：

- PDF：页预览
- 图片：缩略预览
- 文本：内容预览
- Office 文件：先做系统打开或截图预览

不要把第一阶段强依赖在系统级 Quick Look 上。  
mac 可以后续做原生增强，但 MVP 应采用自有预览机制。

## 10. 技术实现建议

### 10.1 技术路线选择

这里有两个方向。

### 路线 A：PySide6 / Qt

优点：

- 本地桌面能力强
- 文件系统操作成熟
- 适合深度系统集成
- 和当前项目技术栈接近

缺点：

- 复杂自由画布开发成本较高
- 节点交互、缩放、预览、局部动画的开发体验不如现代前端
- 后期做复杂节点生态会更重

### 路线 B：桌面壳 + Web 画布

推荐：

- `Tauri + React`
- 画布层：`React Flow / XYFlow` 或定制画布
- 局部高性能渲染：`Canvas / PixiJS`
- PDF：`PDF.js`
- 本地文件能力：Tauri 原生桥接

优点：

- 更适合做 ComfyUI 风格自由画布
- 节点交互实现效率高
- 预览、缩放、拖拽、选择框、多选体验更好
- 后期扩展关系图、PDF 拆页节点、注释节点更容易

缺点：

- 本地文件系统桥接要设计得更严谨
- 需要前端和桌面桥接的双栈能力

### 结论

如果这是一个**全新的独立软件**，更建议采用：

- **Tauri + React + 本地文件桥接**

如果只是为了尽快做出一个内部可用原型，也可以先用：

- **PySide6 + QGraphicsView/QGraphicsScene**

但从长期看，想做到“ComfyUI 式自由度 + 文件系统工作台”，路线 B 更有前景。

## 11. 数据存储建议

### 11.1 文件系统依然是真实源

所有文件、文件夹真实状态以磁盘为准。

### 11.2 画布元数据单独存储

建议每个项目目录旁维护一个隐藏工作目录：

```text
案件目录/
  材料A/
  材料B/
  ...
  .case-canvas/
    project.db
    node_layout.json
    preview_cache/
    thumbs/
```

建议内容：

- 节点位置
- 折叠状态
- 标注信息
- 标签颜色
- 关系连线
- 预览缓存
- PDF 拆分历史

### 11.3 推荐存储组合

- 元数据主存储：`SQLite`
- 节点布局缓存：`JSON`
- 缩略图缓存：本地文件缓存

## 12. 技术难点

以下是当前方案最真实的难点。

### 12.1 画布布局和可读性

案件目录深、文件多，节点数量大。  
如何做到：

- 不乱
- 不卡
- 能看清层级
- 能快速定位

这是核心难点之一。

### 12.2 画布操作与真实文件系统同步

这是第一技术风险点。

场景包括：

- 拖动移动失败
- 同名文件冲突
- 外部程序占用文件
- 外部用户手动修改目录
- 画布与磁盘状态不一致

必须设计可靠的同步与回滚机制。

### 12.3 大 PDF 性能

包括：

- 首屏预览速度
- 缩略图生成
- 分页浏览
- 按页拆分
- 拖拽式重组

这部分会明显影响产品体验。

### 12.4 “自由画布”与“文件系统约束”的冲突

用户会自然地希望像脑图一样任意组织节点。  
但真实文件系统有硬约束：

- 文件只能在一个目录里
- 一个文件不能同时属于两个父节点
- 移动会影响磁盘路径

因此需要明确区分：

- `真实父子结构`
- `业务语义连线`

不能混为一谈。

### 12.5 跨平台预览体验

mac、Windows 对文件预览的原生支持差异很大。  
要做到稳定一致，最终还是需要软件自带预览体系。

## 13. 项目可行性判断

### 13.1 结论

这是一个**完全可实现**的项目。  
但它不是“小功能增强”，而是一个中大型独立产品。

### 13.2 难度级别

可粗略分级为：

- 产品复杂度：高
- 交互复杂度：高
- 技术复杂度：高
- 第一阶段可实现性：高
- 一次性做完全部愿景：低

### 13.3 正确策略

正确策略不是“直接做最终版”，而是：

1. 先做文件系统画布浏览与移动
2. 再做预览与批量整理
3. 再做 PDF 拆分归类
4. 最后做高自由度关系画布

---

## 14. 推荐 MVP 范围

如果现在要启动一个独立项目，建议第一版只做：

- 导入案件目录
- 画布浏览
- 节点折叠展开
- 双击打开
- 空格预览
- 新建文件夹分支
- 文件拖动到分支
- 节点备注
- 基础搜索定位

只要这 9 项做好，产品就已经非常有价值。

## 15. 后续建议

如果后续正式立项，建议先补两份文档：

1. **交互原型文档**
   内容：页面结构、节点样式、拖拽规则、预览区、右键菜单

2. **技术架构文档**
   内容：技术栈选择、文件系统同步机制、缓存结构、PDF 处理链路

在没有原型和技术架构之前，不建议直接开写代码。

---

## 16. 最终结论

这个方向本质上不是“给案件软件加一个脑图标签页”，而是可以独立成立的一个新软件产品：

> 一个以案件材料为对象、以自由画布为工作空间、以真实文件系统为底座、以大 PDF 整理为核心优势的桌面工具。

它的价值非常明确，尤其适合法律行业的材料整理、证据归类、案卷全览和超大 PDF 处理场景。

正确路线应当是：

- **作为独立软件立项**
- **先做文件系统画布 MVP**
- **再扩展到 PDF 工作台和关系语义层**
